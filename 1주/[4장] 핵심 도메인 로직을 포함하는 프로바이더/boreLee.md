- 4.1 프로바이더 종류
    - 비즈니스 로직을 수행하는 역할들을 분리하는 것이 중요하다     
      | 프로바이더       | 역할                                  | 예시                                     | React에 비유                                  |
      |------------------|-----------------------------------|------------------------------------------|--------------------------------------------|
      | **Service**       | 도메인 로직을 처리하는 핵심 계층          | `UserService`, `AuthService`            | 상태 로직 처리 (`useUserService()`)           |
      | **Repository**    | DB 접근을 캡슐화한 계층                  | `UserRepository`, `PostRepository`      | API 호출 로직 (`useUserApi()`)              |
      | **Factory**       | 객체 생성을 책임지는 함수나 클래스         | `ConfigFactory`, `JwtOptionsFactory`    | 커스텀 훅 초기화 시 설정값 만드는 함수               |
      | **Helper / Util** | 공통 기능을 제공하는 유틸성 객체           | `DateHelper`, `HashHelper`              | 유틸 함수 (`utils.ts`)                       |




- 4 (쉬어가는 페이지)
    - 스코프
        - `Nodejs는 멀티 스레드 상태 비저장 모델을 따르지 않습니다`
            - 위 문장을 이해하기 전에 nodejs의 특징을 먼저 알아야한다
            - nodejs의 모델 : `싱글 스레드` , `이벤트 루프` , `비동기`
                - 카페에서 일하는 알바생으로 비유하면
                    - 멀티 스레드 : 손님이 커피를 주문할 때 마다 새 직원(=스레드)을 고용하여 처리한다
                    - nodejs
                    직원은 한명 뿐 `(=싱글 스레드)` 주문이 들어오면 커피추출은 커피머신에게 맞기고 다른 손님의 주문을 계속 받는다 `(=비동기)`
                    커피추출이 끝나면 커피머신이 직원에게 알려서 이어서 작업을 할 수 있게 한다 `(=이벤트 루프)`
            - 즉, nodejs가 전통적인 웹서버처럼 각 요청에 대해 새 스레드를 만들어 상태를 분리하지 않는다
            - 그렇기 때문에 공유 자원(메모리 등)에 접근할 때 상태 관리에 신경을 더 써야한다
        - nestjs에서 스코프 옵션은 비동기+싱글 스레드 환경에서 상태 분리를 명확히 해주기 위해 필요하다
        - 스코프의 종류
            
          | 스코프        | 설명                                                                                      | 사용 시점                                                                                     |
          |---------------|---------------------------------------------------------------------------------------------|-----------------------------------------------------------------------------------------------|
          | `DEFAULT` (기본값)   | 싱글톤(Singleton). 앱 시작 시 한 번만 인스턴스 생성됨.                                       | 대부분의 서비스 (상태 없음 or 공유해도 되는 경우)<br>ex) DB 접근, 외부 API, 유틸 로직 처리        |
          | `REQUEST`     | 요청(Request)마다 새로운 인스턴스 생성됨.                                                      | - 로그인된 사용자 정보를 저장하고 싶을 때<br>- 요청 중간에 어떤 상태를 유지하고 싶을 때            |
          | `TRANSIENT`   | 주입될 때마다 매번 새로운 인스턴스 생성                                                       | - 한 요청 안에서도, 같은 서비스가 여러 개 필요한 경우<br>- 각각 다른 설정, 다른 컨텍스트로 쓰고 싶을 때 |
